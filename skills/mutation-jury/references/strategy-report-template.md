## What Your Score Means

**Mutation score: N%** means N% of generated mutants were killed by your tests.

_Interpret this number in context:_
- A high score on low-risk code (utilities, formatting) matters less than a moderate
  score on financial or auth logic
- Score below 60% in any domain means tests largely execute code without verifying behavior
- Score above 85% in critical domains (payments, auth) is a reasonable initial target
- Score by itself is less useful than score *by module* — a 72% average can hide a
  30% score in your most critical path

## How to Generate a Report for Full Verdict Analysis

Run the appropriate command for your stack and share the output:

**Python (mutmut):**
```bash
mutmut run --paths-to-mutate src/[module]/
mutmut html
# Share: htmlcov/ directory or `mutmut show survived` output
```

**JavaScript/TypeScript (Stryker):**
```bash
npx stryker run
# Share: reports/mutation/mutation.json
```

**Java (PIT):**
```bash
mvn test-compile org.pitest:pitest-maven:mutationCoverage -DtargetClasses=com.example.[package].*
# Share: target/pit-reports/<timestamp>/mutations.xml
```

See `references/tool-format-guide.md` for all supported tools and formats.

## Common Reasons for a Low Mutation Score

Without seeing individual mutants, these are the most common culprits:

**Coverage Theater (most common):** Tests execute code but don't assert on behavior.
A test that calls `apply_discount()` and only checks that no exception was thrown doesn't
verify the return value — arithmetic mutations survive freely.
*Fix:* Audit your assertion density. Every test should assert specific return values
or side effects, not just "it ran."

**Missing boundary tests:** Relational operator mutations (`>` vs `>=`) almost always
survive when no test sends exactly-threshold values.
*Fix:* For every conditional with a numeric threshold, add a test at the boundary value.

**Untested error paths:** Surviving void-call removals often mean error handling isn't tested.
*Fix:* Add tests that exercise failure conditions — invalid inputs, unavailable dependencies,
edge cases.

**Equivalent mutants inflating the denominator:** Some tools count equivalent mutants in
the denominator, suppressing your score artificially.
*Fix:* Run a Verdict Report to identify and formally acquit equivalents — your adjusted
score may be meaningfully higher.
